## 实现要点

### 两步检测流程

```
guard(message):
  // Step 1 — 清洗
  trimmed = message.trim()
  if 空 → 拒答: "请输入您的问题"
  if 长度 > 2000 → 拒答: "问题内容过长"
  if 命中注入模式（HTML标签/SQL关键字/<script>） → 拒答: "输入包含不合规内容"

  // Step 2 — 关键词匹配
  score = 0
  for each (keyword, weight) in 三级词典:
    if message.includes(keyword) → score += weight
  if score >= 3 → 放行
  else → 拒答 + 猜测领域
```

### 关键词词典设计

三级权重覆盖 ~40 个关键词：

| 级别 | 权重 | 示例 | 设计意图 |
|------|------|------|---------|
| 核心词 | 3 | 反应釜、聚合釜、搅拌器、夹套、聚合反应、釜体 | 直接命中＝确定在领域内，一个核心词即可放行 |
| 扩展词 | 2 | 温度测点、冷却水、安全阀、引发剂、PE聚合、间歇反应 | 需要两个扩展词组合才能放行，减少误判 |
| 边缘词 | 1 | 化工设备、反应器、巡检、维护保养、换热 | 自身不足放行，但可作为核心词/扩展词的补充加分 |

阈值设为 3：1 个核心词即可放行，或 2 个扩展词组合，或 1 扩展词 + 1 边缘词。

### 领域猜测

拒答时用关键词匹配猜测用户领域：

```
domainPatterns = [
  { words: ["烹饪","菜","红烧","炒","煮","食材"...], domain: "烹饪" },
  { words: ["代码","编程","函数","bug","算法"...], domain: "编程" },
  { words: ["股票","理财","基金","保险"...], domain: "金融" },
]
默认 → "其他"
```

猜测结果填入拒答模板：`"您的提问涉及{domain}领域，将在后续版本中逐步支持"`。让用户感觉系统"懂了"他问的是什么，而非冷冰冰的拦截。

### 注入检测

三个正则，覆盖最常见的攻击模式：

- `<[^>]*>` — HTML 标签，防 XSS 和 markup 注入到前端
- `(DROP|SELECT|INSERT|DELETE|UPDATE).*(FROM|INTO|TABLE)` — SQL 关键字组合，防 prompt 注入模拟 SQL
- `<script[^>]*>` — script 标签，防直接脚本注入

不是完整的 WAF，只是 MVP 级别的基础过滤。生产环境建议前置专业 WAF / 安全网关。

### 未来扩展（注释预留）

**灰度区 LLM 二次确认**：

当得分在 2（接近阈值但不够）时，可能存在误判：
- 正例误拒："这个设备怎么维护"（没提"反应釜"但上下文在问它）
- 引入 LLM 单次分类，prompt 仅要求输出 YES/NO，成本和延迟都很低

**歧义检测**：

模糊输入如"那个温度怎么样了"不直接拒答，返回澄清反问："请说明是哪个设备、哪个测点的温度？"

## 验收标准

- [x] 设备相关问题（含"反应釜""聚合釜""搅拌器"等）得分 >= 3 放行
- [x] 非设备问题（烹饪、编程等）拒答
- [x] 拒答回复包含领域猜测
- [x] 空输入/超长输入/注入模式被拦截
